# ADR 0003: 项目展示数据用静态 JSON,不调 GitHub API

- **状态**: Accepted
- **日期**: 2026-04-29
- **决策者**: LIYUAN

## Context

「项目展示」页需要列出本人的项目卡片(标题、简介、技术栈、链接)。
数据来源候选:

| 选项 | 优点 | 缺点 |
|---|---|---|
| 静态 JSON 手维护 | 零依赖、构建时确定、可控展示顺序与措辞 | 新项目要手动加,旧项目不会自动消失 |
| 调 GitHub REST API 拉仓库列表 | 自动同步 | 需要鉴权 / 速率限制处理 / 仓库元数据不够丰富(往往要二次加工) / 客户端拉取破坏静态站特性 |
| 构建时调 API 生成静态 JSON | 自动同步 + 仍是静态 | 构建复杂度增加、API 不稳时构建失败 |

实际情况:
- 当前展示项目数 4 个,频率每年新增 1-3 个
- 每个项目要写**精挑措辞的简介**,不能直接用 GitHub repo description
- 想展示的项目不一定都在 GitHub(可能有内部 / 私有的)

## Decision

**用 `src/data/projects.json` 静态维护**。

字段约定(以当前文件为准):
- `title`: 项目名
- `description`: 一句话简介
- `tech`: 技术栈数组
- `repo`: GitHub 链接(可选)
- `live`: 在线访问链接(可选)

页面 `pages/projects.astro` 在构建时 import 并 `.map()` 渲染卡片。

## Consequences

**正面**:
- ✅ 零依赖,构建时确定,不会因第三方 API 故障导致页面空白
- ✅ 简介措辞完全可控,不被 GitHub repo description 字数 / 风格限制
- ✅ 私有 / 非 GitHub 项目也能展示
- ✅ 实现简单,阶段 2 半天搞定

**负面 / 限制**:
- ❌ 新增项目需手动改 JSON + 提交 PR,无法"零操作上线"
- ❌ 项目数若膨胀到几十个,JSON 维护成本上升

**未来变更成本**:
- 🔄 项目数 < 20 个,继续静态维护
- 🔄 项目数 > 20 或频繁更新时,改"构建时调 API 生成 JSON":
  保留静态特性,自动同步,但需要处理 API 故障兜底

## 验证依据

阶段 2 完成(2026-04-29),4 个项目卡片正常渲染,
新增项目实测 < 5 分钟(改 JSON + commit + push 触发自动部署)。
